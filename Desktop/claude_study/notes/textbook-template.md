## Шаблон каждой главы

Каждый раздел внутри главы строится по этой структуре:

```
### [Название концепции]

#### Откуда это взялось и зачем (Eloquent JS стиль)
2-3 предложения: историческая или логическая причина почему эта концепция
существует. Не синтаксис — а смысл.

#### Что это такое
Одно чёткое определение. Потом аналогия из реальной жизни.
Потом ASCII-диаграмма если концепция пространственная (память, стек, поток).

#### Плохой пример → Хороший пример (learn.js.ru стиль)
Сначала показать как делают неправильно (и почему это плохо).
Потом — правильный способ. Студент видит контраст.

#### > Частые заблуждения (Teaching Tech Together стиль)
Блок с типичными ошибками в понимании. Формат:
"Многие думают что X — на самом деле Y, потому что Z."

#### Мини-проверка: предсказать вывод (learn.js.ru стиль)
2-3 коротких вопроса перед практикой:
"Что выведет этот код? Подумай до запуска."
```js
[код]
```
*Ответ: [объяснение]*

#### Задания (Teaching Tech Together — faded examples)
Три уровня:
1. 🟢 Заполни пропуски в готовом решении
2. 🟡 Напиши по образцу
3. 🔴 С нуля по условию

В разборе решений — явные шаги: "Шаг 1: ..., Шаг 2: ..., Шаг 3: ..."
```

---

## Правило: никакой воды

**Каждое предложение должно давать конкретное понимание.** Если предложение убрать — и без него всё понятно, значит оно лишнее.

Признаки воды:
- Общие слова без смысла: "очень важная концепция", "широко используется в программировании"
- Повторение того что только что сказал другими словами без новой информации
- Вводные фразы: "Как мы уже знаем...", "Давайте рассмотрим..."
- Похвала студенту внутри объяснения

Подробность — это НЕ вода:
- Конкретный пример который закрепляет абстрактное правило
- Объяснение "почему" а не только "как"
- Контрпример который предотвращает заблуждение
- Пошаговый разбор сложного места

**Формула:** подробно = конкретно + по делу. Вода = много слов, мало смысла.

## Ключевые принципы из каждого источника

### Из Eloquent JavaScript
1. **Исторический нарратив** — перед каждой темой: откуда взялась концепция
2. **Нормализация трудности** — прямо писать "это выглядит странно, и это нормально"
3. **Формула**: что → зачем → синтаксис → пример → исключения
4. **Эволюция кода** — показывать как "плохой" код превращается в чистый

### Из learn.javascript.ru
1. **Риторические вопросы как заголовки**: "Что такое X?", "Зачем нужен X?", "Когда использовать X?"
2. **Контрастивная структура**: может / не может, правильно / неправильно
3. **Блоки "Важно"** и **"На заметку"** — визуально выделенные, не встроенные в текст
4. **Задания с предсказанием вывода** — до написания кода
5. **ASCII-диаграммы** к ключевым аналогиям

### Из Teaching Tech Together (педагогика)
1. **Сначала ментальная модель, потом синтаксис** — пока нет модели, факты не запоминаются
2. **Faded Examples** — полное решение → с пропусками → с нуля
3. **Labeled subgoals** — в разборах задач называть каждый шаг явно
4. **Показывать заблуждения в тексте** — не только в тестах
5. **Formative assessment** — проверка каждые 2-3 раздела, не только в конце
6. **Notional Machine** — явная модель как JS выполняет код (память, стек вызовов)
7. **Parsons Problems** — задания "расставь строки в правильный порядок"

## Блоки оформления (markdown)

### Блок "Важно"
```
> ⚠️ **Важно:** [текст]
```

### Блок "На заметку"
```
> 💡 **На заметку:** [текст]
```

### Блок "Частые заблуждения"
```
> ❌ **Частые заблуждения:**
> - Многие думают что `X` — на самом деле `Y`, потому что...
> - Кажется что `A` и `B` одно и то же — но разница в...
```

### Блок "Мини-проверка"
```
> 🔍 **Мини-проверка:** что выведет этот код? Подумай до запуска.
> ```js
> [код]
> ```
> <details><summary>Ответ</summary>
> [объяснение]
> </details>
```

### Блок "Нормализация трудности"
```
> Это выглядит странно — так и есть. Большинство новичков
> спотыкаются здесь. Это нормально, дальше станет привычным.
```
