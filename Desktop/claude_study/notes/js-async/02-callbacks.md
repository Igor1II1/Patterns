# Колбэки (Callbacks)

> [[js-fundamentals|<-- Назад к оглавлению JS]]

---

## Зачем эта тема существует?

Колбэки — это **первый способ** работы с асинхронным кодом в JavaScript. До появления промисов (ES6, 2015) колбэки были единственным инструментом для обработки результатов асинхронных операций. Понимание колбэков необходимо, потому что они лежат в основе всех остальных асинхронных паттернов, а также используются во множестве API (события DOM, Node.js, таймеры).

---

## Что такое колбэк?

### Что это такое?

Колбэк (callback) — это **функция, которая передается в другую функцию как аргумент** и вызывается позже, когда произойдет какое-то событие или завершится операция.

Аналогия: ты оставляешь свой номер телефона в ресторане — "позвоните мне, когда столик освободится". Ты не стоишь и не ждешь — ты уходишь по делам. Ресторан **перезванивает** тебе, когда будет готов. Твой номер телефона — это колбэк.

### Как работает?

```js
// Колбэк — это просто функция, переданная как аргумент
function приветствие(имя) {
  console.log("Привет, " + имя + "!");
}

function обработатьПользователя(колбэк) {
  const имя = "Игорь";
  колбэк(имя);  // вызываем переданную функцию
}

обработатьПользователя(приветствие); // "Привет, Игорь!"
```

Колбэки бывают **синхронные** и **асинхронные**:

```js
// СИНХРОННЫЙ колбэк — выполняется сразу
const числа = [1, 2, 3];
числа.forEach(function(n) {
  console.log(n);  // выполняется немедленно, по порядку
});
console.log("после forEach"); // выполнится ПОСЛЕ всех console.log выше

// АСИНХРОННЫЙ колбэк — выполняется позже
setTimeout(function() {
  console.log("позже");  // выполнится ПОСЛЕ "после setTimeout"
}, 1000);
console.log("после setTimeout"); // выполнится СРАЗУ
```

> **Частое заблуждение:** "Колбэк = асинхронность." Нет. Колбэк — это просто функция-аргумент. `forEach`, `map`, `filter` тоже используют колбэки, но они синхронные.

## setTimeout и setInterval

### Что это такое?

**`setTimeout`** — выполняет функцию **один раз** через указанное время (в миллисекундах).
**`setInterval`** — выполняет функцию **повторно** через указанный интервал.

Аналогия: `setTimeout` — будильник (прозвенит один раз). `setInterval` — метроном (тикает постоянно).

### Как работает?

```js
// setTimeout — один раз через 2 секунды
setTimeout(function() {
  console.log("Прошло 2 секунды");
}, 2000);

// setInterval — каждые 3 секунды
let счетчик = 0;
const id = setInterval(function() {
  счетчик++;
  console.log("Тик: " + счетчик);

  if (счетчик >= 5) {
    clearInterval(id);  // остановить после 5 тиков
    console.log("Стоп!");
  }
}, 3000);
```

**Отмена таймеров:**

```js
// setTimeout возвращает идентификатор
const timerId = setTimeout(() => console.log("Не выполнится"), 5000);
clearTimeout(timerId);  // отменяем до выполнения

// setInterval — аналогично
const intervalId = setInterval(() => console.log("Тик"), 1000);
clearInterval(intervalId);  // остановка
```

### Плохой пример -> Хороший пример

```js
// ПЛОХО — забыли очистить интервал
setInterval(function() {
  обновитьДанные();
}, 5000);
// Интервал работает вечно, даже когда уже не нужен!
// Утечка памяти, лишние запросы
```

```js
// ХОРОШО — сохраняем id и очищаем когда не нужно
const intervalId = setInterval(function() {
  обновитьДанные();
}, 5000);

// Когда пользователь уходит со страницы:
function приОтключении() {
  clearInterval(intervalId);
}
```

> **Частое заблуждение:** "`setTimeout(fn, 1000)` гарантирует выполнение ровно через 1 секунду." Нет — это **минимальная** задержка. Если стек занят тяжелым кодом, колбэк будет ждать в очереди дольше.

> **Мини-проверка:** Что выведет этот код и почему?
> ```js
> console.log("А");
> setTimeout(() => console.log("Б"), 0);
> console.log("В");
> ```
> Ответ: `А`, `В`, `Б` — даже с задержкой 0 колбэк проходит через очередь.

## setTimeout с задержкой 0

### Что это такое?

`setTimeout(fn, 0)` — частый паттерн, который означает: "выполни эту функцию как можно скорее, но **после** текущего синхронного кода".

### Как работает?

```js
console.log("1 — синхронно");

setTimeout(() => {
  console.log("2 — из очереди задач");
}, 0);

console.log("3 — синхронно");

// Результат:
// 1 — синхронно
// 3 — синхронно
// 2 — из очереди задач
```

```
Стек:     [log("1")] → [setTimeout] → [log("3")] → пусто
                            ↓
Web API:              [таймер 0мс]
                            ↓
Очередь:                        [log("2")]
                                    ↓
Event Loop:    стек пуст? → да → [log("2")] → стек
```

Зачем это нужно? Иногда нужно отложить выполнение, чтобы дать браузеру перерисовать страницу или обработать другие события.

## Callback Hell (Ад колбэков)

### Что это такое?

Когда асинхронные операции зависят друг от друга (вторая начинается после первой, третья после второй), колбэки вкладываются друг в друга. Это создает **пирамиду смерти** — код, который невозможно читать и поддерживать.

### Как работает?

```js
// Задача: загрузить пользователя, его заказы, детали заказа
загрузитьПользователя(1, function(ошибка, пользователь) {
  if (ошибка) {
    console.log("Ошибка загрузки пользователя");
    return;
  }
  загрузитьЗаказы(пользователь.id, function(ошибка, заказы) {
    if (ошибка) {
      console.log("Ошибка загрузки заказов");
      return;
    }
    загрузитьДетали(заказы[0].id, function(ошибка, детали) {
      if (ошибка) {
        console.log("Ошибка загрузки деталей");
        return;
      }
      показать(детали);
      // ...а если нужна ещё одна вложенность?
    });
  });
});
```

Визуально:
```
загрузить(1, function() {
  ├── загрузить(id, function() {
  │   ├── загрузить(orderId, function() {
  │   │   ├── загрузить(detailId, function() {
  │   │   │   ├── ...ещё глубже...
  │   │   │   │   └── // невозможно читать
  │   │   │   │
  │   │   │   });
  │   │   });
  │   });
  });
```

### Плохой пример -> Хороший пример

```js
// ПЛОХО — Callback Hell
получитьДанные("/api/user", function(err, user) {
  if (err) { обработатьОшибку(err); return; }
  получитьДанные("/api/posts/" + user.id, function(err, posts) {
    if (err) { обработатьОшибку(err); return; }
    получитьДанные("/api/comments/" + posts[0].id, function(err, comments) {
      if (err) { обработатьОшибку(err); return; }
      показать(user, posts, comments);
    });
  });
});
```

```js
// ЛУЧШЕ — вынести колбэки в именованные функции
function показатьКомментарии(err, comments) {
  if (err) { обработатьОшибку(err); return; }
  показать(comments);
}

function обработатьПосты(err, posts) {
  if (err) { обработатьОшибку(err); return; }
  получитьДанные("/api/comments/" + posts[0].id, показатьКомментарии);
}

function обработатьПользователя(err, user) {
  if (err) { обработатьОшибку(err); return; }
  получитьДанные("/api/posts/" + user.id, обработатьПосты);
}

получитьДанные("/api/user", обработатьПользователя);
```

Это лучше, но всё равно неудобно. Настоящее решение — **промисы** (следующая глава).

## Error-First Callbacks (Колбэки с ошибкой первым аргументом)

### Что это такое?

В Node.js сложилась конвенция: первый аргумент колбэка — **всегда ошибка**. Если ошибки нет, первый аргумент равен `null`. Это называется **error-first callback**.

### Как работает?

```js
// Паттерн: callback(error, result)

function читатьФайл(путь, callback) {
  // Имитация асинхронного чтения
  setTimeout(() => {
    if (путь === "") {
      callback(new Error("Путь не может быть пустым"), null);
    } else {
      callback(null, "Содержимое файла: " + путь);
    }
  }, 100);
}

// Использование
читатьФайл("data.txt", function(err, data) {
  if (err) {
    console.log("Ошибка: " + err.message);
    return;  // ВАЖНО: выходим, чтобы не выполнять код ниже
  }
  console.log(data);
});
```

### Плохой пример -> Хороший пример

```js
// ПЛОХО — забыли проверить ошибку
читатьФайл("data.txt", function(err, data) {
  console.log(data);  // если err !== null, data может быть undefined!
});
```

```js
// ХОРОШО — всегда проверяем ошибку первой
читатьФайл("data.txt", function(err, data) {
  if (err) {
    console.error("Не удалось прочитать файл:", err.message);
    return;
  }
  console.log(data);
});
```

> **Частое заблуждение:** "Если я не проверю ошибку в колбэке, программа упадет." Нет — ошибка просто **проигнорируется**, а код продолжит работу с `undefined` данными, что ещё хуже: тихая ошибка, которую трудно найти.

## Практический пример: последовательное выполнение

```js
// Имитация асинхронных операций
function шаг1(callback) {
  setTimeout(() => {
    console.log("Шаг 1 выполнен");
    callback(null, "результат-1");
  }, 1000);
}

function шаг2(данные, callback) {
  setTimeout(() => {
    console.log("Шаг 2 получил: " + данные);
    callback(null, "результат-2");
  }, 1000);
}

function шаг3(данные, callback) {
  setTimeout(() => {
    console.log("Шаг 3 получил: " + данные);
    callback(null, "финальный результат");
  }, 1000);
}

// Последовательное выполнение через колбэки
шаг1(function(err, res1) {
  if (err) return console.error(err);

  шаг2(res1, function(err, res2) {
    if (err) return console.error(err);

    шаг3(res2, function(err, res3) {
      if (err) return console.error(err);

      console.log("Готово: " + res3);
    });
  });
});

// Через 3 секунды (1 + 1 + 1):
// Шаг 1 выполнен
// Шаг 2 получил: результат-1
// Шаг 3 получил: результат-2
// Готово: финальный результат
```

> **Мини-проверка:** Почему в каждом колбэке важно писать `return` после обработки ошибки?
> Ответ: без `return` код продолжит выполняться дальше, и мы попытаемся работать с данными, которых нет.

## Итог

| Понятие | Описание |
|---------|----------|
| **Колбэк** | Функция, переданная как аргумент и вызываемая позже |
| **Синхронный колбэк** | Выполняется сразу (forEach, map) |
| **Асинхронный колбэк** | Выполняется позже (setTimeout, запросы) |
| **setTimeout** | Однократный отложенный вызов |
| **setInterval** | Повторяющийся вызов через интервал |
| **Callback Hell** | Вложенные колбэки — нечитаемый код |
| **Error-First** | Конвенция: `callback(err, data)` — ошибка первая |

**Проблемы колбэков:**
- Callback Hell — глубокая вложенность
- Сложная обработка ошибок (нужно проверять в каждом колбэке)
- Трудно делать параллельные операции
- Нет стандартного способа отменить операцию

Все эти проблемы решают **промисы** (Promise) — тема следующей главы.
