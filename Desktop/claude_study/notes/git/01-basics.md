## Что такое Git и зачем он нужен

**Git** — система контроля версий. Она запоминает все изменения в твоём коде, чтобы ты мог:
- Вернуться к любому предыдущему состоянию проекта
- Понять кто, когда и зачем сделал изменение
- Работать над несколькими функциями одновременно (ветки)
- Сотрудничать с другими разработчиками без конфликтов

**GitHub** — облачное хранилище Git-репозиториев. Git — инструмент, GitHub — платформа.

## Три области Git

```
Рабочая папка        →  Staging Area      →  Репозиторий
(Working Directory)     (Index / Stage)       (Repository)

Редактируешь файлы   →  git add           →  git commit
                        «Хочу сохранить»      «Сохраняю»
```

Это ключевая концепция — файл проходит через **три состояния** прежде чем попасть в историю.

## Начальная настройка (один раз)

```bash
# Представиться Git — кто делает коммиты
git config --global user.name "Твоё Имя"
git config --global user.email "email@example.com"

# Установить редактор (VS Code)
git config --global core.editor "code --wait"

# Проверить настройки
git config --list
```

## Инициализация репозитория

```bash
# Создать новый Git-репозиторий в текущей папке
git init

# Результат: создаётся скрытая папка .git/
# В ней Git хранит всю историю
```

## Статус — что сейчас происходит

```bash
git status
```

Показывает:
- `Untracked files` — новые файлы, Git о них не знает
- `Changes not staged` — изменённые файлы, не добавлены в staging
- `Changes to be committed` — файлы в staging, готовы к коммиту

Запускай `git status` постоянно — это твой основной инструмент навигации.

## git add — добавить в staging

```bash
# Добавить конкретный файл
git add index.js

# Добавить всю папку
git add src/

# Добавить ВСЕ изменения (осторожно — может захватить лишнее)
git add .

# Добавить части файла интерактивно (продвинуто)
git add -p index.js
```

**Зачем staging?** Чтобы делать чистые коммиты. Можно изменить 10 файлов, но закоммитить только 3 — те, что относятся к одной задаче.

---

## git commit — сохранить изменения

```bash
# Базовый коммит
git commit -m "Добавить функцию авторизации"

# Коммит с подробным описанием (открывает редактор)
git commit

# Добавить в staging и закоммитить за один шаг (только уже отслеживаемые файлы)
git commit -am "Исправить баг в форме логина"
```

### Conventional Commits — стандарт сообщений

```bash
# Формат: тип(область): описание

feat: добавить страницу профиля
fix: исправить баг с отображением даты
docs: обновить README
style: отформатировать код
refactor: переписать функцию поиска
test: добавить тесты для API
chore: обновить зависимости
```

**Почему это важно:** через 3 месяца ты откроешь `git log` и поймёшь что происходило без чтения кода.

## .gitignore — что не отслеживать

```bash
# Создай файл .gitignore в корне проекта
```

```gitignore
# Зависимости
node_modules/

# Переменные окружения (НИКОГДА не коммить!)
.env
.env.local
.env.production

# Системные файлы
.DS_Store
Thumbs.db

# Кэш и сборки
dist/
build/
.cache/

# Логи
*.log
npm-debug.log*
```

**Правило:** секреты (пароли, ключи API) — НИКОГДА в Git. Даже если потом удалишь — они остаются в истории.

## git log — история коммитов

```bash
# Полный лог
git log

# Компактный — одна строка на коммит
git log --oneline

# С графом веток
git log --oneline --graph --all

# Последние 5 коммитов
git log -5

# Изменения конкретного файла
git log --oneline index.js

# Поиск по сообщению
git log --grep="авторизация"
```

Пример вывода `git log --oneline`:
```
a3f2c1d feat: добавить поиск по книгам
9b4e2a0 fix: исправить ошибку валидации
3c1d8f2 feat: реализовать добавление книг
f8a9e1b init: начальная структура проекта
```

## git diff — что изменилось

```bash
# Изменения в рабочей папке (не в staging)
git diff

# Изменения в staging (что войдёт в следующий коммит)
git diff --staged

# Разница между двумя коммитами
git diff a3f2c1d 9b4e2a0

# Разница с последним коммитом
git diff HEAD
```

## Типичный рабочий процесс

```bash
# 1. Проверить что сейчас
git status

# 2. Посмотреть изменения
git diff

# 3. Добавить нужные файлы
git add src/library.js

# 4. Проверить что попало в staging
git diff --staged

# 5. Закоммитить
git commit -m "feat: добавить функцию поиска книг"

# 6. Проверить историю
git log --oneline
```

## Частые ошибки

| Ошибка | Что случилось | Решение |
|--------|--------------|---------|
| Закоммитил `.env` | Попали секреты | Срочно в [[git/05-undo]] |
| Коммит с опечаткой | Неправильное сообщение | `git commit --amend -m "новое сообщение"` (только если не запушен) |
| Забыл добавить файл в коммит | Нужный файл не вошёл | `git add file.js` → `git commit --amend --no-edit` |
