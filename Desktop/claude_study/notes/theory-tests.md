### Переменные и типы данных

**Базовый уровень (должен знать перед любым собеседованием):**

**Q1:** В чём разница между `let`, `const` и `var`?
<details>
<summary>Ответ</summary>

- `var` — функциональная область видимости, поднимается с `undefined`, можно повторно объявлять
- `let` — блочная область видимости, поднимается но TDZ (ошибка при обращении до объявления), нельзя повторно объявлять, можно изменять
- `const` — то же что `let`, но нельзя переназначить (объект изменять можно)
- Использовать: `const` везде, `let` когда нужно менять, `var` никогда

</details>

---

**Q2:** Что такое Hoisting (поднятие)?
<details>
<summary>Ответ</summary>

Hoisting — механизм JavaScript при котором объявления переменных и функций "поднимаются" в начало области видимости до выполнения кода.

- `var` — поднимается и инициализируется как `undefined`
- `let`/`const` — поднимаются но НЕ инициализируются (TDZ — обращение = ошибка)
- `function` declaration — поднимается полностью с телом

</details>

---

**Q3:** Что выведет этот код и почему?
```js
console.log(a);
console.log(b);
var a = 1;
let b = 2;
```
<details>
<summary>Ответ</summary>

- `console.log(a)` → `undefined` (var поднялась как undefined)
- `console.log(b)` → `ReferenceError` (let в TDZ — нельзя обращаться до объявления)

</details>

---

**Q4:** Можно ли изменить свойство объекта объявленного через `const`?
<details>
<summary>Ответ</summary>

Да. `const` запрещает **переназначение** переменной, но не запрещает изменять **содержимое** объекта или массива.

```js
const user = { name: "Игорь" };
user.name = "Иван"; // ✅ — меняем свойство
user = {};          // ❌ — переназначаем переменную
```

</details>

---

**Q5:** Сколько типов данных в JavaScript? Назови их.
<details>
<summary>Ответ</summary>

8 типов:
- 7 примитивов: `string`, `number`, `boolean`, `null`, `undefined`, `bigint`, `symbol`
- 1 объектный тип: `object` (включает массивы, функции, объекты)

</details>

---

**Q6:** Что вернёт `typeof null` и почему?
<details>
<summary>Ответ</summary>

`typeof null` → `"object"`

Это историческая ошибка JavaScript из 1995 года которую нельзя исправить без поломки старого кода. `null` — это примитив, а не объект. Правильная проверка на null: `value === null`.

</details>

---

**Q7:** В чём разница между `==` и `===`?
<details>
<summary>Ответ</summary>

- `==` (нестрогое) — перед сравнением приводит типы: `0 == false → true`
- `===` (строгое) — сравнивает и значение и тип: `0 === false → false`

Всегда используй `===`. `==` может дать неожиданные результаты из-за неявного приведения типов.

</details>

---

**Q8:** В чём разница между копированием примитива и объекта?
<details>
<summary>Ответ</summary>

- **Примитивы** — копируются **по значению**: создаётся независимая копия, изменение одной переменной не влияет на другую
- **Объекты** — копируются **по ссылке**: обе переменные указывают на один объект, изменение через одну видно через другую

```js
let a = 5; let b = a; b = 10; // a = 5 ✅
let x = {n:1}; let y = x; y.n = 2; // x.n = 2 ❗
```

</details>

---

**Средний уровень:**

**Q9:** Что такое Temporal Dead Zone?
<details>
<summary>Ответ</summary>

TDZ — промежуток от начала блока до строки объявления `let`/`const`. В этот момент переменная технически поднята, но не инициализирована. Обращение к ней вызывает `ReferenceError`. TDZ защищает от использования переменных до их объявления.

</details>

---

**Q10:** Перечисли "falsy" значения в JavaScript.
<details>
<summary>Ответ</summary>

`false`, `0`, `-0`, `0n`, `""`, `null`, `undefined`, `NaN` — всего 8 значений.

Всё остальное — truthy. Важные исключения: `"0"` (строка) — truthy, `[]` (пустой массив) — truthy, `{}` (пустой объект) — truthy.

</details>

---

**Q11:** В чём разница между `??` и `||`?
<details>
<summary>Ответ</summary>

- `||` — возвращает правый операнд если левый **falsy** (`false`, `0`, `""`, `null`, `undefined`, `NaN`)
- `??` — возвращает правый операнд только если левый `null` или `undefined`

```js
0 || "default"   // → "default"  (0 falsy — || заменяет)
0 ?? "default"   // → 0          (?? не трогает 0, он не null/undefined)

"" || "пусто"    // → "пусто"    ("" falsy)
"" ?? "пусто"    // → ""         ("" не null/undefined)
```

Правило: если `0`, `false`, `""` — **валидные** значения → используй `??`. Иначе можно взять `||`.

</details>

---

**Q12:** Что делает оператор `??=`? Чем он отличается от `||=`?
<details>
<summary>Ответ</summary>

- `??=` — присваивает значение только если переменная `null` или `undefined`
- `||=` — присваивает если переменная falsy

```js
let a = 0;
a ??= 5;   // → 0 (0 не null/undefined — не трогает)
a ||= 5;   // → 5 (0 falsy — заменяет)
```

Используй `??=` для установки дефолтных значений когда `0` и `false` — допустимые настройки.

</details>

---

**Q13:** Почему `typeof NaN === "number"`? Как правильно проверить значение на NaN?
<details>
<summary>Ответ</summary>

`NaN` (Not a Number) — это специальное значение **типа** `number`. Появляется как результат ошибочной числовой операции (`"abc" * 2 → NaN`). Тип числовой — потому что результат должен быть числом, просто невалидным.

```js
typeof NaN          // → "number"  (NaN — это числовой тип)
NaN === NaN         // → false     (NaN не равен сам себе — единственное исключение в JS)
```

Правильная проверка:
```js
Number.isNaN(NaN)   // → true  ✅ (строгая проверка)
isNaN("abc")        // → true  ❌ (сначала конвертирует в число, потом проверяет — неожиданно!)
Number.isNaN("abc") // → false ✅ (строка — не NaN, просто не число)
```

Вывод: всегда используй `Number.isNaN()`, не глобальный `isNaN()`.

</details>

---

**Q14:** Что такое "примитив" в JavaScript? Чем примитивы отличаются от объектов?
<details>
<summary>Ответ</summary>

Примитив — простое значение, хранящееся напрямую в переменной. 7 примитивов:
`string`, `number`, `boolean`, `null`, `undefined`, `bigint`, `symbol`

Всё остальное — объекты: `{}`, `[]`, `function(){}`, `new Map()` и т.д.

Ключевое отличие:
- Примитивы копируются **по значению** — изменение копии не затрагивает оригинал
- Объекты копируются **по ссылке** — обе переменные указывают на один объект

```js
let a = 5; let b = a; b = 10;    // a = 5 (не изменилась)
let x = {n:1}; let y = x; y.n=2; // x.n = 2 (изменилась!)
```

</details>

---

**Q15:** Почему `0.1 + 0.2 !== 0.3`? Как это обойти?
<details>
<summary>Ответ</summary>

JavaScript хранит числа в формате IEEE 754 (64-битное двоичное представление). Некоторые десятичные дроби (0.1, 0.2) не имеют точного представления в двоичном виде — как `1/3` нельзя записать точно в десятичном.

```js
0.1 + 0.2  // → 0.30000000000000004
```

Решения:
```js
// Для отображения:
(0.1 + 0.2).toFixed(2)                    // → "0.30"

// Для сравнения:
Math.abs(0.1 + 0.2 - 0.3) < 1e-10        // → true

// Для финансовых расчётов:
Math.round((0.1 + 0.2) * 100) / 100       // → 0.3
```

</details>

### Функции

**Q11:** Чем Function Declaration отличается от Function Expression?
<details>
<summary>Ответ</summary>

- **Function Declaration** (`function f() {}`) — поднимается полностью, можно вызвать до объявления
- **Function Expression** (`const f = function() {}`) — не поднимается, нельзя вызвать до объявления

</details>

---

**Q12:** В чём главное отличие стрелочной функции от обычной?
<details>
<summary>Ответ</summary>

Главное отличие — **нет своего `this`**. Стрелочная функция берёт `this` из внешнего контекста.

Дополнительно: нет `arguments`, не может быть конструктором (`new`), не имеет `prototype`.

</details>

---

**Q13:** Что такое rest-параметры и чем они отличаются от spread-оператора?
<details>
<summary>Ответ</summary>

Синтаксис одинаковый (`...`), но:
- **Rest** — в **параметрах** функции, **собирает** аргументы в массив: `function f(...args)`
- **Spread** — при **вызове** или в **литерале**, **разворачивает** массив/объект: `Math.max(...arr)`, `[...a, ...b]`

</details>

---

**Q14:** Что такое коллбэк (callback)?
<details>
<summary>Ответ</summary>

Коллбэк — функция переданная как аргумент другой функции, чтобы та вызвала её в нужный момент. Основа асинхронного программирования и методов массивов (`map`, `filter`, `forEach`).

</details>

### Массивы

**Q15:** Чем `map` отличается от `forEach`?
<details>
<summary>Ответ</summary>

- `map` — возвращает **новый массив** с результатами коллбэка
- `forEach` — ничего не возвращает (`undefined`), используется для побочных эффектов

</details>

---

**Q16:** Что делает `reduce`? Объясни на примере.
<details>
<summary>Ответ</summary>

`reduce` сворачивает массив в одно значение. Принимает коллбэк `(acc, current) => newAcc` и начальное значение аккумулятора.

```js
[1,2,3,4,5].reduce((acc, n) => acc + n, 0); // 15
```

На каждом шаге acc получает результат коллбэка. В конце возвращается финальное значение acc.

</details>

---

**Q17:** В чём ловушка метода `sort()` при сортировке чисел?
<details>
<summary>Ответ</summary>

По умолчанию `sort` сравнивает элементы **как строки**. Поэтому `[10, 2, 1].sort()` → `[1, 10, 2]`.

Правильно: `arr.sort((a, b) => a - b)` — числовое сравнение.

Плюс: `sort` **изменяет** исходный массив. Чтобы сохранить оригинал: `[...arr].sort(...)`.

</details>

---

**Q18:** Чем `find` отличается от `filter`?
<details>
<summary>Ответ</summary>

- `find` — возвращает **первый** подходящий элемент (или `undefined`)
- `filter` — возвращает **массив** всех подходящих элементов (или `[]`)

</details>

### Объекты

**Q19:** Когда использовать `obj.key` а когда `obj["key"]`?
<details>
<summary>Ответ</summary>

- `obj.key` — когда имя свойства известно заранее и является допустимым идентификатором
- `obj["key"]` — когда ключ хранится в переменной (`obj[variable]`), содержит спецсимволы или начинается с цифры

</details>

---

**Q20:** Что такое деструктуризация объекта?
<details>
<summary>Ответ</summary>

Синтаксис для извлечения свойств объекта в переменные за одну операцию.

```js
const { name, age = 0 } = user;
```

Поддерживает переименование (`{ name: userName }`), значения по умолчанию, вложенную деструктуризацию, использование в параметрах функций.

</details>

---

**Q21:** Что делает spread оператор с объектом? В чём его ограничение?
<details>
<summary>Ответ</summary>

Spread создаёт **поверхностную копию** объекта. Свойства первого уровня копируются по значению, вложенные объекты — по ссылке.

```js
const copy = { ...original }; // поверхностная копия
```

Ограничение: вложенные объекты по-прежнему общие. Для глубокой копии: `structuredClone(obj)`.

</details>

---

**Q22:** Чем `??` (nullish coalescing) отличается от `||`?
<details>
<summary>Ответ</summary>

- `||` — возвращает правую часть для любого **falsy** значения (включая `0`, `""`, `false`)
- `??` — возвращает правую часть только для `null` или `undefined`

```js
0 || "дефолт"  // "дефолт" (0 — falsy)
0 ?? "дефолт"  // 0 (0 — не null/undefined)
```

Используй `??` когда `0` или `""` — допустимые значения.

</details>

### this

**Q23:** Чему равен `this` в разных ситуациях?
<details>
<summary>Ответ</summary>

- `obj.method()` → `this = obj`
- `function()` (без объекта) → `undefined` (strict mode) или `window`
- `() => {}` (стрелочная) → `this` из внешнего контекста (нет своего)
- `new Class()` → новый экземпляр
- `.call(obj)` / `.apply(obj)` / `.bind(obj)()` → `obj`

</details>

---

**Q24:** Что такое потеря контекста и как её избежать?
<details>
<summary>Ответ</summary>

Потеря контекста — когда метод объекта вызывается без объекта (`const fn = obj.method; fn()`), `this` становится `undefined`.

Решения:
1. Стрелочная обёртка: `() => obj.method()`
2. `bind`: `method.bind(obj)`
3. Стрелочный метод в классе: `tick = () => {}`

</details>

---

**Q25:** Почему нельзя использовать стрелочную функцию как метод объекта?
<details>
<summary>Ответ</summary>

Стрелочная функция берёт `this` из места где она **написана**. Объектный литерал `{}` не создаёт свой контекст — стрелка берёт `this` из внешнего (глобального) контекста, где нет свойств объекта.

```js
const obj = {
  name: "Игорь",
  greet: () => this.name // this = undefined (глобальный), не obj
};
```

</details>

**Q26:** Что такое инкапсуляция? Как реализуется в JS?
<details>
<summary>Ответ</summary>

Инкапсуляция — скрытие внутренних данных объекта и предоставление контролируемого доступа через методы.

**Q27:** Зачем нужен `super()` в конструкторе наследника?
<details>
<summary>Ответ</summary>

`super()` вызывает конструктор родительского класса. Это обязательно — без `super()` в конструкторе наследника нельзя использовать `this` (будет ReferenceError).

`super()` должен быть вызван до любого использования `this`.

</details>

---

**Q28:** Как имитировать абстрактный класс в JavaScript?
<details>
<summary>Ответ</summary>

JS не имеет нативных абстрактных классов. Имитация через `new.target`:

```js
class Abstract {
  constructor() {
    if (new.target === Abstract) {
      throw new Error("Нельзя создать напрямую");
    }
  }
  method() {
    throw new Error("Должен быть переопределён");
  }
}
```

В TypeScript есть ключевое слово `abstract`.

</details>

## Условия (if/else, switch, тернарный)

---

**Q29:** Назови все falsy значения в JavaScript. Что такое truthy?
<details>
<summary>Ответ</summary>

Falsy значений ровно 8: `false`, `0`, `-0`, `0n`, `""`, `null`, `undefined`, `NaN`

Всё остальное — truthy. Важные исключения которые многих удивляют:
- `"0"` — truthy (непустая строка)
- `[]` — truthy (пустой массив)
- `{}` — truthy (пустой объект)

</details>

---

**Q30:** Чем тернарный оператор отличается от `if/else`? Когда его НЕ стоит использовать?
<details>
<summary>Ответ</summary>

Тернарный — краткая запись `if/else` для присвоения значения:
```js
const label = age >= 18 ? "Взрослый" : "Ребёнок";
```

НЕ использовать когда:
- Логика сложная (вложенные тернарные — нечитаемо)
- Нужно выполнить несколько действий, а не выбрать значение
- Условие длинное

</details>

---

**Q31:** Когда использовать `switch` вместо `if/else`?
<details>
<summary>Ответ</summary>

`switch` — когда одна переменная проверяется на конкретные значения (не диапазоны):

```js
switch (role) {
  case "admin": ...; break;
  case "user": ...; break;
}
```

`if/else` — когда нужны диапазоны (`age > 18`), сложные условия (`&&`, `||`), или условий мало (2-3).

Важно: `break` в switch обязателен, без него код "проваливается" в следующий case.

</details>

---

**Q32:** Что выведет код и почему?
```js
console.log(0 || "дефолт");
console.log(0 ?? "дефолт");
console.log(false && "привет");
console.log("user" && "admin");
console.log([] || "пусто");
```
<details>
<summary>Ответ</summary>

- `0 || "дефолт"` → `"дефолт"` (0 — falsy, || берёт правый)
- `0 ?? "дефолт"` → `0` (?? срабатывает только для null/undefined, 0 — не null)
- `false && "привет"` → `false` (&& возвращает первый falsy)
- `"user" && "admin"` → `"admin"` (оба truthy — && возвращает последний)
- `[] || "пусто"` → `[]` ([] — truthy, || берёт первый truthy)

</details>

---

**Q33:** Что такое "короткое замыкание" (short-circuit evaluation)?
<details>
<summary>Ответ</summary>

`&&` и `||` не просто возвращают `true/false` — они возвращают одно из значений и останавливаются как только результат известен:

- `&&` останавливается на первом falsy: `null && doSomething()` — `doSomething` не вызовется
- `||` останавливается на первом truthy: `user.name || "Гость"` — берёт имя если есть

Практически:
```js
isLoggedIn && showMenu();     // вызовет showMenu только если isLoggedIn = true
const name = user.name || "Гость"; // дефолт если имя пустое
```

</details>

---

**Q34:** В чём разница между `??` и `||`? Когда важно использовать `??`?
<details>
<summary>Ответ</summary>

- `||` — возвращает правый операнд для **любого falsy** (0, "", false, null, undefined)
- `??` — возвращает правый операнд только для **null или undefined**

Важно когда 0 или "" — допустимые значения:
```js
const volume = userSettings.volume || 50;  // ❌ если volume = 0 → получим 50
const volume = userSettings.volume ?? 50;  // ✅ если volume = 0 → получим 0
```

</details>

---

**Q35:** Что такое "потенциально опасный" вывод условия?
```js
function getUser(id) {
  if (id) {
    return fetchUser(id);
  }
}
getUser(0);
```
<details>
<summary>Ответ</summary>

`0` — falsy значение, поэтому условие `if (id)` не выполнится даже если `id = 0` — корректный ID.

Правильная проверка:
```js
if (id !== null && id !== undefined) { ... }
// или
if (id != null) { ... }  // != проверяет и null и undefined
```

</details>

### Циклы

---

**Q36:** Назови все виды циклов в JS. Для каждого скажи когда использовать.
<details>
<summary>Ответ</summary>

- `for` — счётчик, нужен индекс, известно кол-во итераций
- `while` — неизвестное кол-во итераций, пока условие true
- `do...while` — как while, но тело выполнится хотя бы раз
- `for...of` — перебор значений итерируемого (массив, строка, Map, Set)
- `for...in` — перебор ключей объекта (не использовать для массивов!)

Для массивов предпочти методы: `forEach`, `map`, `filter` — они декларативнее.

</details>

---

**Q37:** Чем `for...of` отличается от `for...in`? Почему нельзя использовать `for...in` для массивов?
<details>
<summary>Ответ</summary>

- `for...of` — перебирает **значения** итерируемого объекта
- `for...in` — перебирает **ключи** объекта (имена свойств как строки)

Для массива `for...in` вернёт индексы как строки ("0", "1", "2"), а не элементы. Плюс может вернуть унаследованные свойства. Используй `for...of` или `forEach`.

```js
const arr = ["a", "b"];
for (const key in arr) console.log(key);   // "0", "1" — не то!
for (const val of arr) console.log(val);   // "a", "b" ✅
```

</details>

---

**Q38:** Что делают `break` и `continue`? Приведи пример где каждый полезен.
<details>
<summary>Ответ</summary>

- `break` — немедленно выходит из цикла
- `continue` — пропускает текущую итерацию, переходит к следующей

```js
// break — найти первый отрицательный и остановиться:
for (const n of numbers) {
  if (n < 0) { console.log(n); break; }
}

// continue — вывести только нечётные:
for (let i = 0; i < 10; i++) {
  if (i % 2 === 0) continue;
  console.log(i);
}
```

</details>

---

**Q39:** Что выведет код?
```js
for (let i = 0; i < 3; i++) {
  if (i === 1) continue;
  console.log(i);
}

let i = 5;
while (i > 0) {
  i -= 2;
}
console.log(i);
```
<details>
<summary>Ответ</summary>

Первый цикл: `0`, `2` (при i=1 пропускаем через continue)

While: `i` идёт 5 → 3 → 1 → -1. Выведет `-1` (условие i > 0 нарушается когда i = -1, но -= 2 уже выполнилось)

</details>

---

**Q40:** Когда лучше использовать `map/filter/forEach` вместо цикла `for`?
<details>
<summary>Ответ</summary>

Методы массива предпочтительнее когда:
- Нужно трансформировать массив → `map`
- Нужно отфильтровать → `filter`
- Нужно просто пройтись → `forEach`
- Нужно свернуть в одно значение → `reduce`

`for` предпочтительнее когда:
- Нужен `break` (выйти досрочно)
- Нужен доступ к индексу
- Сложная логика с несколькими переменными

</details>

---

**Q41:** В чём разница `while` и `do...while`?
<details>
<summary>Ответ</summary>

- `while` — сначала проверяет условие, потом выполняет. Может не выполниться ни разу.
- `do...while` — сначала выполняет, потом проверяет. Выполнится **хотя бы один раз**.

```js
let i = 10;
while (i < 5) { console.log(i); }    // не выполнится ни разу
do { console.log(i); } while (i < 5); // выполнится один раз
```

Используй `do...while` когда первый запуск нужен в любом случае (например, запросить ввод пользователя).

</details>

---

**Q42:** Что такое "бесконечный цикл"? Как его избежать?
<details>
<summary>Ответ</summary>

Бесконечный цикл — цикл условие которого никогда не становится `false`. Программа зависает.

```js
// ❌ Бесконечный — забыли изменить i:
let i = 0;
while (i < 10) { console.log(i); } // i никогда не меняется!

// ✅ Правильно:
while (i < 10) { console.log(i); i++; }
```

Правило: в каждом цикле должна быть строка которая **приближает** к завершению условия.

</details>

**Q43:** Почему строки называют "неизменяемыми"? Что это означает на практике?
<details>
<summary>Ответ</summary>

Неизменяемость (immutability) означает что **каждый метод строки возвращает новую строку**, не изменяя оригинал.

```js
const str = "привет";
str.toUpperCase(); // вернул "ПРИВЕТ"
console.log(str);  // "привет" — оригинал не изменился!

// Чтобы "изменить" — нужно перезаписать переменную:
const upper = str.toUpperCase();
```

</details>

**Q44:** Чем шаблонные литералы лучше конкатенации через `+`?
<details>
<summary>Ответ</summary>

```js
// Конкатенация — неудобно, легко ошибиться:
"Привет " + name + ", тебе " + age + " лет"

// Шаблонный литерал — читаемо:
`Привет ${name}, тебе ${age} лет`
```

Преимущества:
- Читаемость — видно структуру строки
- Многострочность — просто нажать Enter
- Выражения внутри `${}` — любой JS код

</details>

**Q45:** Чем `indexOf` отличается от `includes`? Когда использовать каждый?
<details>
<summary>Ответ</summary>

- `includes(str)` → `true/false` — просто проверить наличие
- `indexOf(str)` → число (позиция) или `-1` — нужна позиция

```js
email.includes("@");       // true/false — для проверки ✅
email.indexOf("@");        // 4 — нужна позиция @ ✅
```

Используй `includes` когда нужна просто проверка — это явнее и читаемее.

</details>

**Q46:** Как работает `slice`? Как вырезать последние N символов?
<details>
<summary>Ответ</summary>

`slice(начало, конец)` — вырезает подстроку. `конец` не включается. Поддерживает отрицательные индексы (с конца).

```js
"abcde".slice(1, 3);  // "bc"
"abcde".slice(2);     // "cde" — от 2 до конца
"abcde".slice(-2);    // "de" — последние 2 символа
"abcde".slice(0, -1); // "abcd" — всё кроме последнего
```

</details>

**Q47:** Как разбить строку на массив и собрать обратно? Приведи практический пример.
<details>
<summary>Ответ</summary>

- `str.split(разделитель)` → массив
- `arr.join(разделитель)` → строка

```js
"яблоко,банан,груша".split(","); // ["яблоко", "банан", "груша"]
["яблоко", "банан"].join(" и "); // "яблоко и банан"

// Практически — сделать kebab-case:
"hello world".split(" ").join("-"); // "hello-world"
```

</details>

**Q49:** Назови 5 самых полезных методов строк которые используются каждый день.

### Обработка ошибок

---

**Q50:** Зачем нужен `try/catch`? Что происходит без него?
<details>
<summary>Ответ</summary>

Без `try/catch` — при ошибке программа полностью останавливается. Весь код ниже не выполнится.

`try/catch` позволяет "поймать" ошибку и продолжить работу программы, обработав исключительную ситуацию.

```js
// Без try/catch — программа падает:
const data = JSON.parse("плохой JSON"); // ❌ всё, дальше не работает

// С try/catch — программа продолжает:
try {
  const data = JSON.parse("плохой JSON");
} catch (e) {
  console.log("Невалидный JSON"); // обработали
}
// код продолжается ✅
```

</details>

---

**Q51:** Что такое объект ошибки? Какие у него свойства?
<details>
<summary>Ответ</summary>

Объект ошибки `error` в блоке `catch` содержит:
- `error.name` — тип ошибки (`"TypeError"`, `"ReferenceError"`, etc.)
- `error.message` — текстовое описание
- `error.stack` — стек вызовов (для отладки)

```js
try {
  null.property;
} catch (error) {
  console.log(error.name);    // "TypeError"
  console.log(error.message); // "Cannot read properties of null"
}
```

</details>

---

**Q52:** Зачем нужен `finally`? Когда он выполняется?
<details>
<summary>Ответ</summary>

`finally` выполняется **всегда** — и при успехе, и при ошибке. Используется для очистки ресурсов которую нужно сделать в любом случае.

Типичные случаи:
- Скрыть индикатор загрузки
- Закрыть соединение с БД
- Разблокировать UI

```js
try {
  showSpinner();
  const data = await fetchData();
  displayData(data);
} catch (e) {
  showError(e.message);
} finally {
  hideSpinner(); // скроем в любом случае
}
```

</details>

---

**Q53:** Чем `throw new Error("msg")` отличается от `throw "msg"`?
<details>
<summary>Ответ</summary>

`throw "строка"` — бросает строку, у неё нет `.name`, `.message`, `.stack`. Трудно обрабатывать.

`throw new Error("msg")` — бросает объект Error с полным набором свойств. Можно:
- Проверить тип через `instanceof`
- Получить стек вызовов
- Создать кастомный класс ошибки

```js
// ❌ Плохо:
throw "что-то пошло не так";

// ✅ Хорошо:
throw new Error("что-то пошло не так");
throw new TypeError("ожидалось число");
```

</details>

---

**Q54:** Назови встроенные типы ошибок в JS и когда каждый возникает.
<details>
<summary>Ответ</summary>

- `TypeError` — неверный тип: `null.prop`, `undefined()`, вызов не-функции
- `ReferenceError` — необъявленная переменная: `console.log(x)` без объявления x
- `SyntaxError` — неверный синтаксис: `JSON.parse("{bad}")`
- `RangeError` — значение вне диапазона: `new Array(-1)`

</details>

---

**Q55:** Зачем создавать кастомные классы ошибок? Покажи пример.
<details>
<summary>Ответ</summary>

Кастомные ошибки позволяют:
- Различать типы ошибок через `instanceof`
- Добавлять дополнительные поля (какое поле невалидно, код статуса)
- Давать понятные имена бизнес-ошибкам

```js
class ValidationError extends Error {
  constructor(field, message) {
    super(message);
    this.name = "ValidationError";
    this.field = field;
  }
}

try {
  throw new ValidationError("email", "Некорректный email");
} catch (e) {
  if (e instanceof ValidationError) {
    console.log(`Ошибка в поле ${e.field}`); // "Ошибка в поле email"
  }
}
```

</details>

**Q56:** Зачем нужны модули? Какие проблемы они решают?
<details>
<summary>Ответ</summary>

Без модулей весь код в одном файле — это:
- Тяжело читать и поддерживать
- Конфликты имён переменных
- Невозможно переиспользовать код

Модули дают:
1. **Организацию** — каждый файл отвечает за одно
2. **Изоляцию** — переменные внутри модуля не доступны снаружи (без экспорта)
3. **Переиспользование** — написал один раз, импортируй где угодно
4. **Явные зависимости** — видно что откуда берётся

</details>

**Q57:** Чем именованный экспорт отличается от дефолтного?
<details>
<summary>Ответ</summary>

**Именованный** — можно несколько из одного файла, имя фиксировано:
```js
export function add() {}
export const PI = 3.14;
// импорт: import { add, PI } from "./math.js"
```

**Дефолтный** — один на файл, имя при импорте любое:
```js
export default class User {}
// импорт: import User from "./user.js"
// или:   import MyUser from "./user.js"  — имя любое
```

</details>

---

**Q58:** Когда использовать `default export`, а когда `named export`?
<details>
<summary>Ответ</summary>

`default export` — когда в файле **одна главная вещь**: класс, компонент React, главная функция.

`named export` — когда в файле **несколько утилит** одной темы: `formatDate`, `formatPrice`, `formatName` в одном `format.js`.

Правило: один файл — одна ответственность. Если файл называется `User.js` — логично дефолтный экспорт. Если `utils.js` — именованные.

</details>

---

**Q59:** Что такое barrel файл (`index.js`)? Зачем он нужен?
<details>
<summary>Ответ</summary>

Barrel файл реэкспортирует всё из папки через один `index.js`:

```js
// components/index.js
export { default as Button } from "./Button.js";
export { default as Header } from "./Header.js";

// Теперь вместо:
import Button from "./components/Button.js";
import Header from "./components/Header.js";

// Можно так:
import { Button, Header } from "./components";
```

Удобно для организации кода в больших проектах.

</details>

---

**Q60:** Что такое динамический импорт? Зачем он нужен?
<details>
<summary>Ответ</summary>

Динамический `import()` загружает модуль **по требованию**, а не при старте приложения.

```js
// Статический — грузится всегда:
import { heavyChart } from "./chart.js";

// Динамический — грузится только когда нужен:
button.addEventListener("click", async () => {
  const { drawChart } = await import("./chart.js");
  drawChart(data);
});
```

Используется для:
- Ленивой загрузки (code splitting) в Next.js/React
- Условного импорта
- Уменьшения начального размера приложения

</details>

**Q61 — Замыкания**
**Вопрос:** Что такое замыкание? Приведи простой пример.
**Ответ:** Замыкание — это функция, которая "помнит" переменные из внешней области видимости даже после того, как внешняя функция завершила работу. Пример:
```js
function makeCounter() {
  let count = 0;           // эта переменная "захвачена" замыканием
  return () => ++count;
}
const counter = makeCounter();
counter(); // 1
counter(); // 2 — count сохраняется между вызовами
```
**Почему это важно:** Замыкания — основа приватных данных, фабричных функций, коллбэков и паттернов типа модуль.

---

**Q62 — Замыкания**
**Вопрос:** Почему возникает классический баг с замыканием в цикле с `var`? Как исправить?
**Ответ:** `var` имеет функциональную область видимости, поэтому все итерации цикла замыкаются на одну и ту же переменную `i`, которая к моменту вызова функций уже равна конечному значению.
```js
// Баг:
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0); // выводит 3, 3, 3
}
// Исправление через let (блочная область видимости):
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0); // выводит 0, 1, 2
}
```
**Почему это важно:** Классический вопрос на собеседовании.

---

**Q63 — Замыкания**
**Вопрос:** Что такое IIFE и зачем он нужен?
**Ответ:** IIFE (Immediately Invoked Function Expression) — функция, которая вызывается сразу после объявления:
```js
(function() {
  const private = "недоступна снаружи";
  // код изолирован
})();
```
Использовался до ES6 модулей для создания приватной области видимости. Сейчас актуален в редких случаях (одноразовая инициализация, изоляция полифилла).
**Почему это важно:** Понимание IIFE показывает знание истории JS и паттернов изоляции кода.

---

**Q64 — Прототипы**
**Вопрос:** Что такое прототипная цепочка? Как JS ищет метод у объекта?
**Ответ:** Каждый объект в JS имеет внутреннюю ссылку `[[Prototype]]` на другой объект (прототип). При обращении к свойству JS ищет его:
1. На самом объекте
2. Если нет — у его прототипа
3. Если нет — у прототипа прототипа
4. И так до `Object.prototype`, после которого `null`
```js
const arr = [1, 2, 3];
// arr.map → не на arr → ищет в Array.prototype → нашёл!
```
**Почему это важно:** Понимание прототипов объясняет как работают классы, наследование и почему методы не дублируются в каждом объекте.

---

**Q65 — Прототипы**
**Вопрос:** В чём разница между `__proto__` и `.prototype`?
**Ответ:**
- `__proto__` — ссылка на прототип конкретного объекта (у любого объекта): `obj.__proto__`
- `.prototype` — свойство функции-конструктора, которое становится `__proto__` для создаваемых экземпляров: `Array.prototype`

```js
function Dog() {}
const rex = new Dog();

rex.__proto__ === Dog.prototype; // true
Dog.prototype !== Dog.__proto__; // они разные!
```
**Почему это важно:** Путаница между ними — частая ошибка на собеседовании.

---

**Q66 — Прототипы**
**Вопрос:** Как работает `instanceof`?
**Ответ:** `instanceof` проверяет, есть ли `Constructor.prototype` в прототипной цепочке объекта:
```js
[] instanceof Array;   // Array.prototype в цепочке [] → true
[] instanceof Object;  // Object.prototype тоже в цепочке → true
{} instanceof Array;   // Array.prototype НЕТ в цепочке {} → false
```
**Почему это важно:** Объясняет почему `[] instanceof Object === true` и как работает проверка типов.

---

**Q67 — Продвинутые классы**
**Вопрос:** Что такое `static` метод? В чём разница между статическим и обычным методом?
**Ответ:** Static метод принадлежит классу, а не его экземплярам. Вызывается через класс, не через `new`:
```js
class MathUtils {
  static add(a, b) { return a + b; }
  multiply(a, b) { return a * b; } // обычный — нужен экземпляр
}

MathUtils.add(2, 3);       // ✅ 5
new MathUtils().multiply(2, 3); // ✅ 6

MathUtils.multiply(2, 3);  // ❌ не функция
new MathUtils().add(2, 3); // ❌ не функция
```
**Почему это важно:** Static используется для утилитарных функций, фабричных методов, счётчиков экземпляров.

---

**Q68 — Продвинутые классы**
**Вопрос:** Что делают геттеры (get) и сеттеры (set)?
**Ответ:** Позволяют использовать синтаксис свойства (`obj.prop`) вместо вызова метода (`obj.getProp()`), но при этом выполнять логику:
```js
class Circle {
  constructor(radius) { this._radius = radius; }

  get area() {                           // геттер — вычисляет при обращении
    return Math.PI * this._radius ** 2;
  }

  set radius(value) {                    // сеттер — валидирует при присвоении
    if (value <= 0) throw new Error("Радиус должен быть > 0");
    this._radius = value;
  }
}

const c = new Circle(5);
c.area;      // 78.54... — вызов геттера как свойство!
c.radius = -1; // Error!
```
**Почему это важно:** Геттеры/сеттеры позволяют добавлять валидацию и вычисления сохраняя чистый API.

**Q69 — JSON**
**Вопрос:** Что теряется при JSON.stringify()? Почему?
**Ответ:** JSON поддерживает только базовые типы: строка, число, булево, null, объект, массив. Всё остальное теряется или изменяется:
```js
JSON.stringify({
  fn: function() {},  // → исчезает
  sym: Symbol(),      // → исчезает
  undef: undefined,   // → исчезает
  nan: NaN,           // → null
  inf: Infinity,      // → null
  date: new Date(),   // → строка ISO
});
// '{"nan":null,"inf":null,"date":"2024-01-15T..."}'
```
**Почему это важно:** Знание ограничений JSON критично при работе с API и localStorage.

**Q70 — JSON**
**Вопрос:** Как безопасно работать с JSON.parse()? Что может пойти не так?
**Ответ:** `JSON.parse()` бросает `SyntaxError` при невалидном JSON:
```js
// Небезопасно:
const data = JSON.parse(userInput); // упадёт если userInput невалидный JSON

// Безопасно:
function safeParse(str) {
  try {
    return JSON.parse(str);
  } catch {
    return null;
  }
}
```
Частые ошибки: одинарные кавычки, trailing commas, undefined как значение.
**Почему это важно:** Данные от пользователя или сервера могут быть невалидными.

**Q71 — Map и Set**
**Вопрос:** Когда использовать Map вместо обычного объекта?
**Ответ:** Map предпочтительнее когда:
1. Ключи не строки (объекты, числа, функции)
2. Важен порядок вставки
3. Часто добавляются/удаляются элементы (Map оптимизирован)
4. Нужно знать размер — `map.size` vs `Object.keys(obj).length`
```js
// Объект как ключ — только в Map!
const cache = new Map();
const key = { userId: 1 };
cache.set(key, { data: "..." });
cache.get(key); // работает!
```
**Почему это важно:** Map vs Object — частый вопрос о правильном выборе структуры данных.

**Q72 — Map и Set**
**Вопрос:** Как Set помогает убрать дубликаты из массива?
**Ответ:** Set автоматически хранит только уникальные значения. Для удаления дубликатов:
```js
const arr = [1, 2, 2, 3, 3, 3];

// Способ 1:
const unique = [...new Set(arr)]; // [1, 2, 3]

// Способ 2:
const unique2 = Array.from(new Set(arr)); // [1, 2, 3]
```
Set использует строгое равенство (`===`), поэтому с объектами не работает как ожидается.
**Почему это важно:** Классический паттерн удаления дубликатов, знакомый всем JS разработчикам.

**Q73 — RegExp**
**Вопрос:** Что означают символы: `^`, `$`, `\d`, `\w`, `+`, `*`, `?` в регулярных выражениях?
**Ответ:**
- `^` — начало строки: `/^hello/` совпадает с "hello world" но не "say hello"
- `$` — конец строки: `/world$/` совпадает с "hello world" но не "world peace"
- `\d` — любая цифра [0-9]
- `\w` — буква, цифра или _: [a-zA-Z0-9_]
- `+` — один или более: `/\d+/` → "123"
- `*` — ноль или более: `/\d*/` → "" или "123"
- `?` — ноль или один: `/colou?r/` → "color" и "colour"
**Почему это важно:** Базовый синтаксис RegExp необходим для валидации, парсинга и поиска.

**Q74 — RegExp**
**Вопрос:** В чём разница между `match()` и `matchAll()`?
**Ответ:**
```js
const str = "cat bat hat";

str.match(/[a-z]at/);     // ["cat"] — первое совпадение
str.match(/[a-z]at/g);    // ["cat", "bat", "hat"] — все (флаг g)
[...str.matchAll(/[a-z]at/g)]; // [{0:"cat", index:0}, {0:"bat"...}] — с метаданными
```
`matchAll` возвращает итератор с полной информацией о каждом совпадении (индекс, группы).
**Почему это важно:** `matchAll` нужен когда важна информация о позиции или именованных группах.

**Q75 — Date и Math**
**Вопрос:** Почему месяцы в Date начинаются с 0? Как правильно создать дату "15 января 2024"?
**Ответ:** Это историческое наследие C/Java. Месяц в конструкторе Date — индекс массива (0=январь, 11=декабрь):
```js
new Date(2024, 0, 15);  // 15 января 2024 ✅
new Date(2024, 1, 15);  // 15 февраля 2024
new Date("2024-01-15"); // альтернатива через ISO строку — без смещения!
```
**Почему это важно:** Классический источник багов с датами.

**Q76 — Date и Math**
**Вопрос:** Почему `0.1 + 0.2 !== 0.3` в JavaScript?
**Ответ:** Числа с плавающей точкой в JS (IEEE 754) не могут точно представить 0.1 и 0.2 в двоичном формате, поэтому при сложении накапливается погрешность:
```js
0.1 + 0.2; // 0.30000000000000004

// Решения:
Math.round((0.1 + 0.2) * 100) / 100 === 0.3; // true
Math.abs(0.1 + 0.2 - 0.3) < Number.EPSILON;   // true
```
**Почему это важно:** Один из самых известных "подводных камней" JS. Важен при работе с деньгами и точными вычислениями.

**Q77 — Итераторы и генераторы**
**Вопрос:** Что такое итерируемый объект? Какие встроенные типы итерируемы?
**Ответ:** Итерируемый объект — любой объект с методом `[Symbol.iterator]()`, который возвращает итератор. Итератор — объект с методом `next()` возвращающим `{ value, done }`.

Встроенные итерируемые типы: Array, String, Map, Set, NodeList, arguments, generator.
```js
const str = "abc";
for (const char of str) console.log(char); // a b c
const [a, b] = "hi"; // деструктуризация тоже использует итератор
```
**Почему это важно:** Понимание протокола итерации объясняет как работает `for...of`, spread (`...`), деструктуризация.

**Q78 — Итераторы и генераторы**
**Вопрос:** Что такое генератор? Чем `yield` отличается от `return`?
**Ответ:** Генератор (`function*`) — функция, которую можно приостанавливать и возобновлять. `yield` приостанавливает генератор возвращая значение, `return` завершает его:
```js
function* gen() {
  yield 1; // пауза, вернёт { value: 1, done: false }
  yield 2; // пауза, вернёт { value: 2, done: false }
  return 3; // конец, вернёт { value: 3, done: true }
}

const g = gen();
g.next(); // { value: 1, done: false }
g.next(); // { value: 2, done: false }
g.next(); // { value: 3, done: true }
g.next(); // { value: undefined, done: true }
```
**Почему это важно:** Генераторы — основа ленивых вычислений и лежат в основе async/await.
