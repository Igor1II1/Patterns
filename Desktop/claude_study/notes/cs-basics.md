## Как компьютер выполняет код

### Что такое CPU

**CPU (Central Processing Unit)** — процессор, "мозг" компьютера. Он умеет делать одно: выполнять инструкции. Очень быстро, по одной.

Инструкции процессора — это примитивные команды:
- сложить два числа
- скопировать данные из одного места памяти в другое
- перейти к другой инструкции

```
CPU понимает только числа. Всё остальное — текст, картинки, код — это числа в памяти.
```

### Оперативная память (RAM)

**RAM (Random Access Memory)** — оперативная память, "рабочий стол" компьютера. Здесь хранятся данные пока программа работает.

```
Жёсткий диск (SSD/HDD)          RAM                    CPU
─────────────────────────        ──────────────         ──────────
│ Файлы, программы       │ →→→  │ Загруженные  │ →→→  │ Считает │
│ (постоянно)            │      │ данные       │      │ быстро  │
│ Медленно               │      │ (временно)   │      │         │
─────────────────────────        ──────────────         ──────────
```

Когда запускаешь программу — файл грузится с диска в RAM. Когда закрываешь — RAM очищается.

### Машинный код

Процессор понимает только **машинный код** — последовательность байт (чисел от 0 до 255). Человек такое читать не может.

```
Машинный код (байты в hex):
B8 01 00 00 00    ← "поместить число 1 в регистр EAX"
05 02 00 00 00    ← "сложить с числом 2"
```

Именно поэтому были изобретены языки программирования — чтобы писать код в человекочитаемом виде.

## Уровни языков программирования

Языки делятся по тому, насколько они близки к машинному коду.

### Низкоуровневые языки

Близки к машинному коду. Ты управляешь каждым байтом памяти вручную.

```
Ассемблер (почти машинный код, но читаемый):
MOV EAX, 1    ; поместить 1 в регистр
ADD EAX, 2    ; сложить с 2

C:
int x = 1 + 2;    // проще, но всё равно управляешь памятью вручную
```

**Плюсы:** максимальная скорость, полный контроль над железом
**Минусы:** сложно, долго писать, код зависит от конкретного процессора
**Где используют:** операционные системы (Linux ядро — на C), драйверы, встроенные системы

### Высокоуровневые языки

Далеко от машинного кода. Многое делается автоматически.

```js
// JavaScript — высокоуровневый:
const users = ["Игорь", "Анна", "Петя"];
const result = users.filter(name => name.length > 4);

// За этой одной строкой скрываются тысячи машинных инструкций
```

**Плюсы:** быстро писать, читаемо, переносимо между платформами
**Минусы:** медленнее низкоуровневых (но для большинства задач хватает)
**Где используют:** веб, мобильные приложения, бизнес-логика

### Шкала уровней

```
НИЗКИЙ уровень          Средний                  ВЫСОКИЙ уровень
(близко к железу)                                (далеко от железа)
──────────────────────────────────────────────────────────────────
Машинный     Ассемблер    C/C++    Java/C#    Python/JS/Ruby
код
```

JavaScript — высокоуровневый язык. Именно поэтому на нём удобно начинать — не нужно думать о памяти, байтах, регистрах.

## Компилируемые vs Интерпретируемые языки

Как код из файла становится инструкциями для CPU?

### Компилируемые языки

**Компилятор** — программа, которая читает весь твой код и **заранее** переводит его в машинный код. Результат — исполняемый файл (.exe на Windows).

```
твой код (C++)  →  компилятор  →  program.exe  →  запускаешь
         (один раз)                              (работает быстро)
```

**Примеры:** C, C++, Go, Rust
**Плюс:** очень быстро работает
**Минус:** нужно компилировать под каждую платформу отдельно

### Интерпретируемые языки

**Интерпретатор** — программа, которая читает твой код и выполняет его **прямо сейчас**, строка за строкой.

```
твой код (Python)  →  интерпретатор  →  выполняет на лету
                   (читает и выполняет одновременно)
```

**Примеры:** Python, Ruby
**Плюс:** переносимо (один код — любая платформа с интерпретатором)
**Минус:** медленнее компилируемых

### JavaScript — особый случай (JIT)

JavaScript формально интерпретируемый, но современные движки (V8 в Chrome и Node.js) используют **JIT-компиляцию** (Just-In-Time).

```
твой JS код  →  V8 интерпретирует  →  горячий код  →  компилирует
                (начало работы)       (часто вызывается) (ускоряет)
```

JIT — компромисс: гибкость интерпретации + скорость компиляции. Поэтому JS на V8 работает значительно быстрее "чистых" интерпретируемых языков.

## Система типов

Один из самых важных аспектов любого языка программирования — как он работает с типами данных.

### Статическая vs Динамическая типизация

**Статическая типизация** — тип переменной задаётся при объявлении и не меняется. Ошибки типов находит **компилятор до запуска**.

```typescript
// TypeScript (статический):
let age: number = 25;
age = "двадцать пять"; // ❌ Ошибка на этапе компиляции — ещё до запуска!
```

**Динамическая типизация** — тип переменной определяется при выполнении и может меняться. Ошибки типов обнаруживаются **во время работы программы**.

```js
// JavaScript (динамический):
let age = 25;         // сейчас number
age = "двадцать пять"; // ✅ OK — тип изменился, JS не против
age = true;           // ✅ OK — снова изменился
```

**Плюсы динамики:** быстрее писать, гибче
**Минусы динамики:** ошибки типов обнаруживаются только при запуске

> **К чему ты придёшь:** TypeScript — это статически типизированный JavaScript. После изучения JS (Блок 0) перейдёшь на TS, который добавляет статическую типизацию поверх JS.

### Строгая vs Слабая типизация

**Строгая типизация** — нельзя автоматически смешивать разные типы. Нужно явно конвертировать.

```python
# Python — строгий:
"5" + 2      # ❌ TypeError: can only concatenate str (not "int") to str
int("5") + 2 # ✅ 7 — явное преобразование
```

**Слабая типизация** — язык автоматически преобразует типы чтобы операция "сработала".

```js
// JavaScript — слабый:
"5" + 2    // "52" — число стало строкой (конкатенация)
"5" - 2    // 3   — строка стала числом (вычитание)
"5" * 2    // 10  — снова строка стала числом
```

JavaScript — **слабо типизированный** язык. Это удобно, но создаёт неожиданные результаты:

```js
[] + []    // "" (пустая строка)
[] + {}    // "[object Object]"
{} + []    // 0 (в некоторых контекстах!)
```

Именно поэтому в JS так важно понимать преобразование типов.

### Явное vs Неявное преобразование типов

**Неявное (implicit)** — JS сам, тихо, конвертирует тип:

```js
// JS конвертирует автоматически:
if ("hello") { ... }    // строка → boolean (truthy)
5 + "5"                 // число → строка → "55"
true + 1                // boolean → число → 2
```

Это происходит без твоего ведома — именно это делает слабую типизацию "слабой".

**Явное (explicit)** — ты сам, осознанно, конвертируешь:

```js
Number("42")    // → 42     (строка → число)
String(42)      // → "42"   (число → строка)
Boolean(0)      // → false  (число → boolean)

parseInt("42px")    // → 42 (убирает "px")
parseFloat("3.14")  // → 3.14
```

Явное преобразование — лучше. Ты контролируешь что происходит.

### Где JavaScript на этой шкале

```
                    Типизация по времени проверки:
                    Статическая          Динамическая
                    ──────────           ────────────
Строгость  Строгая │ Java, C++          │ Python
типизации          │                    │
           Слабая  │ (редко)            │ JavaScript ← здесь
```

JavaScript — **динамически и слабо типизированный** язык. TypeScript добавляет статическую типизацию поверх (но остаётся слабым).

## IDE — что это и зачем

### Редактор кода vs IDE

**Редактор кода (Text editor)** — просто программа для редактирования текста с подсветкой синтаксиса. Блокнот с цветами.

**IDE (Integrated Development Environment)** — "умная" среда разработки, которая понимает твой код.

```
Текстовый редактор:              IDE (VS Code с расширениями):
────────────────────             ──────────────────────────────
✅ Подсветка синтаксиса          ✅ Подсветка синтаксиса
                                 ✅ Автодополнение кода
                                 ✅ Переход к определению функции
                                 ✅ Поиск всех использований
                                 ✅ Рефакторинг (переименование)
                                 ✅ Встроенный отладчик (debugger)
                                 ✅ Интеграция с Git
                                 ✅ Встроенный терминал
                                 ✅ Расширения (плагины)
```

### Что умеет VS Code

**VS Code** — технически редактор, но с расширениями становится полноценной IDE:

```
Autocomplete (IntelliSense):
  Начинаешь писать "cons" → выпадает "console.log" → Tab → вставилось

Go to Definition (F12):
  Нажал F12 на функции → перешёл к её объявлению (даже в другом файле)

Find All References (Shift+F12):
  Где ещё используется эта переменная? VS Code покажет все места

Refactor → Rename Symbol (F2):
  Переименовать переменную сразу во всём проекте

Debugger:
  Поставить точку остановки (breakpoint) → запустить → смотреть значения
  переменных в реальном времени. Мощнее чем console.log!

Terminal (Ctrl+`):
  Встроенный терминал — не нужно переключаться между окнами
```

### Как IDE понимает код

IDE использует **Language Server Protocol (LSP)** — отдельная программа которая анализирует твой код и отвечает на вопросы:

```
VS Code                         Language Server (для JS — это tsserver)
──────────                      ────────────────────────────────────────
"Пользователь пишет 'arr.'"  → "Вот методы массива: push, pop, map..."
"Он нажал F12 на 'getUser'"  → "Функция определена в api/users.js:42"
"Что за ошибка тут?"         → "Переменная 'x' не объявлена"
```

Для JS/TS эту работу делает **TypeScript Language Server** — даже когда ты пишешь обычный JS!

### Расширения которые стоит установить

```
Для JavaScript/TypeScript:
  ESLint             — находит ошибки и стилевые проблемы
  Prettier           — форматирует код автоматически
  GitLens            — показывает кто и когда изменил строку

Для веба:
  Live Server        — автообновление браузера при сохранении файла
  CSS Peek           — видеть CSS по классу прямо в HTML

Общие:
  Path Intellisense  — автодополнение путей к файлам
  Error Lens         — ошибки прямо в строке кода (не только снизу)
```

## Как это всё связано: от кода до CPU

Проследим путь от твоего JS файла до выполнения на процессоре:

```
Ты пишешь:
  const x = 1 + 2;
      ↓
  script.js (UTF-8 текст)
      ↓
  Node.js / Браузер запускает V8 (движок JavaScript)
      ↓
  V8: Парсинг → Abstract Syntax Tree (дерево кода)
      ↓
  V8: Компиляция в байткод (промежуточный формат)
      ↓
  V8: JIT — "горячий" код компилируется в машинный код
      ↓
  CPU выполняет машинные инструкции
  (сложить 1 и 2, записать в память, ...)
      ↓
  RAM: x = 3 (4 байта по адресу, например, 0x7FFF5ABC)
```

Ты никогда не работаешь с этим напрямую — V8 делает всё сам. Но понимать что "под капотом" происходит именно это — полезно.

## Итог

```
УРОВНИ ЯЗЫКОВ:
  Низкоуровневые (Assembler, C) → максимум контроля и скорость
  Высокоуровневые (JS, Python)  → удобство, скорость разработки
  JavaScript — высокоуровневый

КАК ВЫПОЛНЯЕТСЯ КОД:
  Компилируемые → сначала компиляция, потом exe-файл (C, Go)
  Интерпретируемые → выполняется на лету (Python)
  JS → V8 с JIT (гибрид, очень быстрый)

ТИПИЗАЦИЯ:
  Статическая  — типы проверяются до запуска (TypeScript, Java)
  Динамическая — типы проверяются во время работы (JavaScript)

  Строгая — нельзя смешивать типы без явной конвертации (Python)
  Слабая  — автоматически конвертирует типы (JavaScript)

  Явное преобразование:   Number("42"), String(42), Boolean(0)
  Неявное преобразование: "5" + 2 → "52" (JS делает сам)

JavaScript = динамический + слабый
TypeScript  = статический + слабый (следующий шаг после JS)

IDE vs РЕДАКТОР:
  Редактор — просто цветной текст
  IDE — понимает код, автодополнение, рефакторинг, отладка
  VS Code + расширения = IDE для JS/TS разработки

ОТ КОДА К CPU:
  JS файл → V8 парсит → байткод → JIT → машинный код → CPU
```
